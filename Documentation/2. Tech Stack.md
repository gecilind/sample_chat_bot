## Component-by-Component Breakdown

---

### 1. AI Brain — OpenAI

**Tech Stack:**
- **OpenAI Assistants API** — chat reasoning, intent classification, function calling, RAG
- **OpenAI Realtime API** — in-app voice (Phase 2)

**Phase 1 (Chat):** Assistants API handles text reasoning, function calling (ticket creation), and RAG via our pgvector knowledge base.

**Phase 2 (Voice):** Realtime API handles in-app voice. Browser captures audio via `getUserMedia()` + `AudioWorklet`, streams PCM16 frames over WebSocket through our `/voice-relay` Python route to OpenAI's Realtime API. No phone numbers. No telephony.

---

### 2. Backend — Python + FastAPI

All business logic runs in a Python + FastAPI server. This is our code — fully owned, version controlled in Git, deployable to any server (Railway, Render, VPS, etc.).

**Our Routes:**

| Route | Purpose | What It Calls |
|---|---|---|
| `POST /chat` | Receives user message, calls OpenAI Assistants API, returns response | OpenAI API + Supabase Postgres (conversation history) |
| `POST /create-ticket` | Creates a support ticket from AI-structured data | Infleet's Jira REST API via `httpx` |
| `WS /voice-relay` | Phase 2: WebSocket relay between browser and OpenAI Realtime API | OpenAI Realtime API |
| `POST /ingest-manual` | Chunks documents, generates embeddings, stores in pgvector | OpenAI Embeddings API + Supabase Postgres |
| `POST /search-kb` | Internal backend endpoint for RAG search. The OpenAI `search_knowledge_base` function tool maps to this handler inside our backend process. It is not publicly exposed to the frontend. | Supabase Postgres pgvector query |

---

### 3. Database — Supabase PostgreSQL + pgvector

Supabase is used **only as a hosted Postgres database**. Our Python backend connects to it like any standard Postgres database. No Edge Functions, no auth, no dashboard logic.

**Embedding model: `text-embedding-3-small` → `vector(1536)` — locked.** All ingestion code, the `manuals` table schema, and the HNSW index must use 1536 dimensions. Do not use `text-embedding-3-large` (3072 dims) — changing this after the index is built requires a full re-ingestion.

**What we store in Supabase:**
- Conversation history
- Manual embeddings — `vector(1536)` column
- Ticket references

**pgvector — RAG Query:**

```sql
SELECT
  id,
  source,
  section,
  content,
  1 - (embedding <=> query_embedding) AS similarity
FROM manuals
WHERE category = 'hardware'
ORDER BY embedding <=> query_embedding
LIMIT 5;
```

**HNSW Index (required from Day 1 — without it pgvector does a full table scan):**

```sql
CREATE INDEX ON manuals USING hnsw (embedding vector_cosine_ops)
WITH (m = 16, ef_construction = 64);
```

---

### 4. Frontend — React + Vite (our code)

**Components:**
- **Custom React Chat Component** — Phase 1, full branding control
- **Custom React Voice Widget** — Phase 2, `getUserMedia()` + `AudioWorklet`
- **Chat/Voice Selection Screen** — entry component shown before conversation starts
- **User context reader** — reads `window.__INFLEET_USER__` from Infleet's session on mount, no login UI on our side

All API calls go to our Python backend via native `fetch()` or WebSocket. No third-party SDK needed on the frontend.

**There is no auth UI in our frontend.** The user is already logged in via Infleet's portal before our widget loads.

---

### 5. Infleet Login System (client's existing authentication)

- Infleet maintains their own portal authentication entirely
- User logs in through their existing system before our widget is ever loaded
- Infleet passes user context to our React component as injected page variables
- We read that context and pass it to our Python backend — we don't create, validate, or manage sessions

**What We Receive:**
```typescript
// Injected by Infleet after their own login — provides user identity only.
// Device identification happens in conversation (AI asks for the device serial/vehicle name).
window.__INFLEET_USER__ = {
  userId: "infleet_user_abc123",
  email: "john@example.com",
};
```

---

### 6. Jira Integration (Infleet's own infrastructure)

Our Python backend calls Jira directly via `httpx` — no SDK, no integration platform.

```python
# Inside /create-ticket route
async with httpx.AsyncClient() as client:
    response = await client.post(
        f"{os.environ['JIRA_BASE_URL']}/rest/api/3/issue",
        headers={"Authorization": f"Basic {credentials}"},
        json={"fields": {"project": {"key": JIRA_PROJECT_KEY}, ...}}
    )
```

## Architecture Summary

| Capability | Implementation |
|---|---|
| User Login / Authentication | **Infleet's existing portal** (we consume the session — we don't build or touch it) |
| User Context Handoff | **`window.__INFLEET_USER__`** injected by Infleet after their login |
| AI Chat Reasoning | **OpenAI Assistants API** (direct, no middleman) |
| AI Voice | **OpenAI Realtime API** (in-app only, no telephony) |
| Knowledge Base / RAG | **pgvector on Supabase PostgreSQL** |
| Backend Logic | **Python + FastAPI** (our code, version controlled, deployable anywhere) |
| Ticket Creation | **`/create-ticket` route → Jira `httpx` call** (direct, no SDK) |
| Chat UI | **Custom React component** (full ownership) |
| Voice UI | **Custom React + AudioWorklet** (Phase 2, full ownership) |
| Database | **Supabase PostgreSQL** (storage only — conversations, embeddings) |
| **Total external vendors** | **2 (OpenAI + Supabase)** |
